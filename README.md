# Docs Hub

Docs Hub — self-hosted база знаний для команды: Markdown-статьи, права доступа, группы, вложения, поиск, резервные копии и простой деплой одним Go-бинарём.

Проект подходит для внутренней документации, регламентов, заметок по проектам, runbook-ов, onboarding-материалов и закрытых справочников.

## Что умеет Docs Hub

- хранит статьи в Markdown с live preview;
- поддерживает `[[wiki-links]]`, `#tags` и backlinks;
- показывает пользователю только те статьи, к которым у него есть доступ;
- позволяет выдавать доступ всем авторизованным, отдельным пользователям или группам;
- даёт администраторам интерфейс для пользователей, групп, статей, аудита и backup-ов;
- поддерживает вложения к статьям с проверкой прав при скачивании;
- создаёт ZIP-backup с `storage.json` и `uploads/`;
- автоматически мигрирует старый файл данных при запуске;
- запускается локально, в Docker или как systemd-сервис за reverse proxy.

## Быстрый старт

Нужен Go 1.23 или новее.

Запуск веб-сервиса:

```bash
ADMIN_USER=admin \
ADMIN_PASSWORD='замени-на-длинный-пароль' \
DATA_FILE='./data/storage.json' \
ADDR=':8080' \
go run . serve
```

Откройте в браузере:

```text
http://localhost:8080
```

Если `ADMIN_USER` и `ADMIN_PASSWORD` не заданы, Docs Hub создаст начального администратора:

```text
admin / admin123
```

Этот пароль нужен только для первого запуска. Сразу замените его на сильный пароль.

## Запуск через Docker

```bash
docker compose up -d --build
```

После запуска приложение будет доступно на:

```text
http://localhost:8080
```

Данные сохраняются в локальной папке `data/`:

```text
data/storage.json
data/uploads/
data/backups/
```

Остановить сервис:

```bash
docker compose down
```

## Сборка бинарного файла

```bash
go test ./...
go build -o docs-hub .
./docs-hub serve
```

Для проверки гонок данных:

```bash
go test -race ./...
```

## Вход и роли

В Docs Hub есть две роли:

- `admin` — управляет пользователями, группами, статьями, вложениями, аудитом и backup-ами;
- `user` — читает доступные статьи и использует поиск.

Сессии хранятся на сервере, cookie содержит только идентификатор сессии и случайный токен. Для административных действий используется CSRF-защита.

## Работа со статьями

Администратор может создавать и редактировать статьи через веб-интерфейс.

Поддерживается обычный Markdown:

```markdown
# Заголовок

Текст, списки, таблицы, кодовые блоки.

Ссылка на другую статью: [[start]]
Ссылка с подписью: [[start|Стартовая статья]]
Тег: #onboarding
```

Редактор использует Toast UI Editor через CDN. Если CDN недоступен, Docs Hub автоматически переключается на обычное поле `textarea`.

## Права доступа

Для каждой статьи можно выбрать аудиторию:

- все авторизованные пользователи;
- конкретные пользователи;
- группы пользователей.

Поиск, список статей, backlinks, preview и выдача вложений учитывают эти права. Пользователь не увидит статью или файл, если у него нет доступа к связанной статье.

## Вложения

Загружать вложения может администратор. Файлы хранятся в `uploads/` под сгенерированными именами, а оригинальное имя сохраняется в метаданных.

Разрешённые расширения:

```text
png, jpg, jpeg, gif, webp, svg, pdf, txt, md, csv, json, zip, docx, xlsx, pptx
```

Лимит одного файла: 25 MB.

## Поиск

Поиск работает по доступным пользователю статьям и учитывает:

- заголовок;
- slug;
- текст статьи;
- теги.

Результаты ранжируются: совпадения в заголовке и slug получают больший вес, чем совпадения в тексте.

## Backup и восстановление

В админке откройте раздел `Backups` и нажмите `Создать backup`. Архив включает:

```text
storage.json
uploads/*
```

Для восстановления:

1. Остановите Docs Hub.
2. Распакуйте backup в каталог данных.
3. Запустите Docs Hub снова.

Скрипт восстановления:

```bash
scripts/restore.sh backup.zip ./data
```

## Deploy TUI

Если запустить приложение без аргументов в интерактивном терминале, откроется TUI-меню деплоя:

```bash
go run .
```

Явный запуск TUI:

```bash
go run . deploy
```

Меню умеет:

- автоматически определить путь к бинарю на сервере: текущий executable, `PATH`, локальный `./docs-hub`, `dist/docs-hub-linux-amd64` или стандартный `/opt/docs-hub/docs-hub`;
- автонастроить деплой на Linux-сервере: создать пользователя `docshub`, рабочие каталоги, установить бинарь, env-файл, systemd unit, watchdog timer и Caddyfile;
- если TUI запущен без root-прав или на Windows, подготовить deploy-файлы и вывести готовые `sudo`-команды для сервера;
- настроить название сайта, домен, email Let's Encrypt, пути, порт и admin-пользователя;
- сгенерировать `deploy/docs-hub.env`, systemd unit, watchdog timer, Caddyfile и deploy compose-файл;
- проверить `/healthz`, состояние systemd и файл данных;
- перезапустить systemd-сервис, если health check не проходит;
- подготовить Caddy-конфигурацию для автоматического HTTPS.

Для production-запуска всегда используйте режим сервера:

```bash
./docs-hub serve
```

## Переменные окружения

| Переменная | Значение по умолчанию | Назначение |
| --- | --- | --- |
| `ADDR` | `:8080` | Адрес и порт HTTP-сервера |
| `DATA_FILE` | `storage.json` | Путь к JSON-файлу данных |
| `UPLOAD_DIR` | рядом с `DATA_FILE` | Каталог вложений |
| `ADMIN_USER` | `admin` | Логин первого администратора |
| `ADMIN_PASSWORD` | `admin123` | Пароль первого администратора |
| `DOCS_HUB_MODE` | пусто | Установите `server`, чтобы не открывать deploy TUI |

`MINIVAULT_MODE=server` пока тоже поддерживается для совместимости со старыми окружениями.

## Production checklist

- задайте сильный `ADMIN_PASSWORD` до первого запуска;
- храните `data/` на persistent volume;
- поставьте Caddy или Nginx перед приложением и включите HTTPS;
- не публикуйте порт приложения напрямую в интернет без reverse proxy и firewall;
- регулярно выгружайте backup-архивы за пределы сервера;
- ограничьте доступ к серверу и файлу `storage.json`;
- после восстановления backup-а проверьте `/healthz` и вход администратора.
